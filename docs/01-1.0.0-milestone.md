# ArchUnit Lens 1.0.0 Milestone

## 목적

이 문서는 기존 `01` 구현 노트, `02` 초기 릴리즈 로드맵, `03` 확장형 rule engine 설계를 하나로 합친 1.0.0 마일스톤 기준 문서다. ArchUnit Lens의 목표는 ArchUnit 테스트를 대체하지 않고, 정적으로 신뢰 가능한 subset을 IntelliJ live inspection으로 먼저 보여주는 것이다.

- `[ ]` `@AnalyzeClasses` scope와 `.because(...)` metadata 지원

- `[ ]` `ArchRuleDefinition` entry point 기준 Subject taxonomy 도입
- `[ ]` target ArchUnit version 기준으로 DSL entry point / Given / Should API를 로컬 source에서 검토
- `[ ]` predicate `and()`/`or()` boolean tree 지원
- `[ ]` 대표 annotation exclusivity rule 지원
- `[ ]` QueryMapper류 naming/interface rule의 핵심 subset 지원
- `[ ]` Proxy annotation rule처럼 `classes()`와 `methods()` subject를 함께 쓰는 meta-annotation rule 후보 지원. 단, custom `DescribedPredicate` helper 해석은 post-1.0으로 보류한다.
- `[ ]` 성능상 대형 프로젝트에서 전체 Java 재스캔 빈도 축소
- `[ ]` 사용자가 플러그인이 무엇을 감지/미감지하는지 알 수 있는 UX 제공
- `[ ]` `check`, `ktlintCheck`, `verifyPlugin`, 대표 fixture 통과

### B. 확장 가능한 Rule Engine 구조

메서드별 `parseXxxRule()`을 계속 추가하지 않는다. 1.0.0 전에는 다음 구조로 전환한다.

```text
ArchRuleSourceFinder
  -> RuleSource(@ArchTest field, containing class, @AnalyzeClasses metadata)
MethodCallChain
  -> RawCall(name, args, source)
RuleNormalizer
  -> RuleDescriptor(scope, subject, predicateExpr, conditionExpr, reason, supportStatus)
InspectionEvaluator
  -> scope filter
  -> predicateExpr evaluation
  -> conditionExpr diagnostics / quick fixes
```

권장 IR:

```kotlin
data class RuleDescriptor(
    val subject: SubjectKind,
    val scope: AnalyzeScope,
    val predicate: PredicateExpr,
    val condition: ConditionExpr,
    val reason: String?,
    val supportStatus: SupportStatus,
)
```

#### Subject taxonomy

ArchUnit DSL은 `ArchRuleDefinition` entry point부터 큰 축이 갈린다. 1.0.0 구조는 이 축을 먼저 보존하고, 각 subject별 predicate/condition registry를 붙인다.

```kotlin
sealed interface SubjectKind {
    data object Classes : SubjectKind
    data object SingleClass : SubjectKind
    data object Members : SubjectKind
    data object Fields : SubjectKind
    data object CodeUnits : SubjectKind
    data object Constructors : SubjectKind
    data object Methods : SubjectKind
    data class CustomTransformer(val description: String?) : SubjectKind
}
```

초기 지원은 `Classes`에 집중한다. 다만 `members()`, `fields()`, `methods()`, `constructors()`, `codeUnits()`, `all(transformer)`를 파서가 만났을 때는 무시하지 말고 `Unsupported(subject = ...)` metadata로 남긴다. 이렇게 해야 tool window/overview에서 “발견했지만 아직 지원하지 않는 rule”을 설명할 수 있다.

`noClasses()`, `noMethods()` 같은 `no*` entry point는 별도 subject가 아니라 같은 subject에 `conditionPolarity = Negated` 또는 root condition negation을 붙이는 방식으로 모델링한다. ArchUnit 소스도 `no*`를 transformer 설명 + condition negation으로 다루므로, Lens도 “대상 종류”와 “조건 부정”을 분리하는 편이 확장에 유리하다.

predicate는 단순 리스트가 아니라 expression tree로 둔다.

```kotlin
sealed interface PredicateExpr {
    data object All : PredicateExpr
    data class Leaf(val predicate: ClassPredicate) : PredicateExpr
    data class And(val left: PredicateExpr, val right: PredicateExpr) : PredicateExpr
    data class Or(val left: PredicateExpr, val right: PredicateExpr) : PredicateExpr
    data class Not(val inner: PredicateExpr) : PredicateExpr
}
```

- `[ ]` `and()`/`or()`는 pending operator로 정규화
- `[ ]` traversal 순서, precedence, 괄호 의미를 확정할 수 없는 chain은 unsupported 처리
- `[ ]` condition은 1차로 `And` 조합만 지원하고, `Or` condition은 false positive 위험을 줄인 뒤 추가
- `[ ]` unsupported call은 나중에 overview에서 보여줄 수 있도록 metadata로 보존
- `[ ]` subject별 handler registry를 둔다. 예: `ClassPredicateHandler`, `MemberPredicateHandler`, `MethodConditionHandler`
- `[ ]` target ArchUnit source를 기준으로 handler 후보를 고른다. 웹 검색보다 `build/reference-sources/archunit/<version>`의 `lang/syntax` package를 우선 탐색한다.
- `[ ]` method/member subject는 `MembersThat.areDeclaredInClassesThat()`처럼 nested class predicate builder를 반환하는 DSL을 별도 node로 표현한다.

#### Custom predicate policy — post-1.0 deferred

ArchUnit은 `DescribedPredicate<T>`와 custom `ArchCondition<T>`를 허용하므로 전체 정적 해석은 불가능하다. Lens는 arbitrary lambda를 실행하지 않는다. 대신 다음처럼 정적으로 안전한 패턴만 제한적으로 해석한다.

이 항목은 현재 1.0.0 첫 구현 slice에서는 제외한다. 1.0.0에서는 custom predicate를 만나면 `SupportStatus.Unsupported(customPredicate)`로 보존하고 warning은 만들지 않는다.

- `[ ]` argument가 local static method call이면 method body를 PSI로 resolve한다.
- `[ ]` `DescribedPredicate.describe("...", lambda)` 형태에서 description을 metadata로 추출한다.
- `[ ]` lambda가 static final `Class<?>[]` 또는 literal class list를 참조하고, `anno.getRawType().isAssignableTo(type)` 같은 단순 membership/assignability pattern이면 class literal set으로 정규화한다.
- `[ ]` 이 범위를 벗어나면 `Unsupported(customPredicate)`로 표시하고 warning은 만들지 않는다.

### C. 성능 / 인덱싱

현재 구조는 inspection 실행 시 `ArchRuleProjectService.rules()`가 lazy하게 rule을 요청하고, PSI modification count가 바뀌면 프로젝트 Java 파일을 다시 훑을 수 있다. 1.0.0에서는 다음으로 개선한다.

- `[ ]` ArchUnit rule source 후보 파일만 빠르게 찾는 index 도입
- `[ ]` 변경된 rule source 파일만 재파싱하는 증분 cache
- `[ ]` `AnalyzeScope` 기준 rule lookup으로 현재 파일에 관련 없는 rule filter 비용 축소
- `[ ]` dumb mode / indexing 중 동작 정책 명확화
- `[ ]` 대형 fixture로 최초 scan, 재검사, 파일 변경 후 재검사 비용 측정

### C-1. ArchUnit target reference

ArchUnit DSL 해석 정확도는 target ArchUnit version에 의존한다. 플러그인 컴파일/런타임에 ArchUnit을 참조하지 않되, 로컬 분석 속도와 정확도를 위해 sources JAR만 별도 작업으로 풀어둔다.

- `[x]` target reference version은 `gradle.properties`의 `archUnit.reference.version`으로 고정한다.
- `[x]` `unpackArchUnitReferenceSources`는 `com.tngtech.archunit:archunit:<version>:sources@jar`만 resolve한다.
- `[x]` output은 `build/reference-sources/archunit/<version>`이며 Git 추적 대상이 아니다.
- `[ ]` 1.0.0에서 지원하는 DSL handler 목록은 이 source tree의 `com/tngtech/archunit/lang/syntax` 기준으로 결정한다.
- `[ ]` version upgrade 시 마일스톤 문서에 “API 차이 / handler 영향 / 테스트 영향”을 기록한다.

## 사용자 예시 매핑

### Mapper annotation exclusivity

```java
classes()
    .that()
    .areAnnotatedWith("org.apache.ibatis.annotations.Mapper")
    .should()
    .notBeAnnotatedWith("io.indoorplus...SecondaryMapper")
    .because("Primary와 Secondary Datasource의 책임을 명확히 분리한다.");
```

IR:

- scope: `AnalyzeClasses(packages = "io.indoorplus")`
- predicate: `AnnotatedWith(org.apache.ibatis.annotations.Mapper)`
- condition: `MustNotBeAnnotatedWith(io.indoorplus...SecondaryMapper)`
- reason: `.because(...)`

### QueryMapper naming/interface rule

```java
classes()
    .that()
    .haveSimpleNameEndingWith("QueryMapper")
    .should()
    .beInterfaces()
    .andShould()
    .beAssignableTo(QueryMapper.class);
```

IR:

- predicate: `NameEndsWith(QueryMapper)`
- condition: `And(MustBeInterface, MustBeAssignableTo(QueryMapper))`

`beAssignableTo`는 PSI resolve와 inheritance 확인이 필요하므로 `beInterfaces` 이후 단계로 구현한다.

### Proxy annotation on interfaces/methods

```java
classes()
    .that()
    .areInterfaces()
    .should()
    .notBeMetaAnnotatedWith(proxyAnnotations())
    .because("프록시 기반 어노테이션은 구체 클래스에만 선언한다.");

methods()
    .that()
    .areDeclaredInClassesThat()
    .areInterfaces()
    .should()
    .notBeMetaAnnotatedWith(proxyAnnotations())
    .because("프록시 기반 어노테이션은 구체 클래스의 메서드에만 선언한다.");
```

IR:

- first rule subject: `Classes`
- first rule predicate: `ClassPredicate.IsInterface`
- second rule subject: `Methods`
- second rule predicate: `MethodPredicate.DeclaredInClass(PredicateExpr.Leaf(IsInterface))`
- shared condition: `MustNotBeMetaAnnotatedWith(AnnotationPredicate.ProxyAnnotations)`
- shared reason: `.because(...)`

`proxyAnnotations()`처럼 `DescribedPredicate<JavaAnnotation<?>>`를 반환하는 helper는 실행하지 않는다. 1.0.0에서는 unsupported metadata로 보존하고 warning은 만들지 않는다. 정적으로 안전한 subset 추출은 post-1.0에서 검토한다.
